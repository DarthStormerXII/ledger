## 0G Storage

### KV vs Log

**Two layers, stacked:**

- **Log layer** — append-only decentralized log for unstructured raw data. Permanent, content-addressed via Merkle root hash. Think IPFS but with on-chain commitment via the Flow contract.
- **KV layer** — structured database **built on top of the log**. Each independent KV DB has a unique **streamId** (32-byte hex, e.g. `0x...f2bd`). KV nodes monitor flagged log entries, deserialize, and replay operations into a local KV state.

For agent memory, **KV is the right primitive** — you write `set(streamId, key, value)` and reads return the latest value for that key. Mutability + millisecond reads.

### Permission model

This is the critical-and-poorly-documented bit. From the storage-kv repo and architecture docs:

- **Writes are gated by transaction submission** — to write to a stream, you submit a tx through the Flow contract paying for storage. Anyone holding the streamId secret and OG to pay can write.
- **Reads are NOT gated** — KV nodes serve queries to anyone. Confidentiality comes from **client-side AES-256-CTR encryption** at upload time (`UploadOption.EncryptionKey`).
- **Permissioning binds to the encryption key, NOT to a wallet or token.** If you hold the symmetric key, you read; if not, ciphertext is opaque.

**This is the architectural pivot point for Ledger.** 0G Storage does NOT enforce token-bound access. The iNFT contract + oracle MUST manage key custody and re-issue keys on transfer. Our oracle holds (or sealed-rewraps) the streamId's encryption key and re-encrypts it for the new owner on `transfer()`.

### How CIDs work

Stored objects are addressed by **Merkle root hash** (not IPFS CIDs). The storage client computes the Merkle tree client-side, submits root via the Flow contract, and the root is the canonical handle. KV layer adds a streamId prefix; individual keys within a stream are addressed `(streamId, key)`.

### Token-bound access — does it exist?

**No, not natively. We build it.** The pattern: agent's memory streamId is encrypted with key K. K is sealed (RSA-OAEP or AES-wrap) under owner's public key and stored on-chain in the iNFT's `_encryptedURIs` mapping. On transfer, oracle TEE unwraps with old owner's key, rewraps under new owner's key, contract updates mapping. **This is the entire iNFT value proposition.** If we don't do this, our submission is a sticker.

## ERC-7857

### Spec status

- **Original GitHub spec:** `docs.0g.ai/developer-hub/building-on-0g/inft/erc7857` (0G's hosted draft).
- **EIP repo:** Search results referenced `github.com/ethereum/EIPs/pull/7857` but that PR resolved to a different EIP (#6123 by cfries, redirected to `ethereum/ERCs`). **The 0G iNFT proposal is NOT yet a numbered ERC in `ethereum/ERCs`.** Treat as **0G-published draft standard**, not Ethereum-canonical.
- **Reference impl:** `github.com/0glabs/0g-agent-nft`, branch `eip-7857-draft`. 16 stars, active. Solidity 52% / TypeScript 48%.

In submission language: write **"ERC-7857 (0G iNFT draft standard)"** — accurate and judge-defensible. Calling it "the ERC-7857 standard" without qualification will draw nitpicks.

### What transfers with the token (precise list)

From the spec text and reference contract:

| Transferred on `transfer()` | Mechanism |
|---|---|
| **tokenId ownership** | Standard ERC-721 `_transfer()` |
| **Encrypted metadata access** | Oracle re-encrypts → contract updates `_encryptedURIs[tokenId]` and `_metadataHashes[tokenId]` |
| **Sealed key for new owner** | `sealedKey` parameter, written to event `MetadataUpdated(tokenId, newHash)` |
| **Authorization grants** | Cleared/reissued depending on impl — `_authorizations[tokenId][address]` |

What does NOT auto-transfer: the underlying encrypted bytes on 0G Storage (they don't move, the **key** moves).

### Reference implementation — key contracts

Branch: `github.com/0glabs/0g-agent-nft/tree/eip-7857-draft`

- **Contracts dir** (`contracts/`): main `IERC7857`/`AgentNFT` implementation. Tracks `_metadataHashes`, `_encryptedURIs`, `_authorizations`, `oracle` address, `PROOF_VALIDITY_PERIOD = 1 hours`.
- **Oracle interface:** delegates `validProof(proof)` to a configurable oracle contract.
- **Mandatory function:**
  ```solidity
  function transfer(
    address from, address to, uint256 tokenId,
    bytes calldata sealedKey, bytes calldata proof
  ) external nonReentrant validProof(proof);
  ```
- **Optional functions:** `clone()` (mint copy with same metadata access), `authorizeUsage()` (delegate without transfer).
- **Events:** `MetadataUpdated(uint256 indexed tokenId, bytes32 newHash)`, `UsageAuthorized`, `OracleUpdated`.
- **Crypto requirements (security-binding):** AES-256-GCM for metadata, RSA-4096 or ECC-P384 for key sealing.

### "Encrypted intelligence transfer" — defensible? how?

**Defensible IF and ONLY IF you ship the oracle.** The full chain:

1. Mint: agent memory (KV stream) is encrypted with key K. K is sealed under owner pubkey, hash committed on-chain.
2. Transfer call: sender provides `sealedKey` (K rewrapped for receiver) + `proof` (TEE attestation over the rewrap operation).
3. Contract calls `oracle.verifyProof(proof)` — verifies the attestation came from a known TEE measurement, with correct inputs/outputs.
4. On success: ERC-721 ownership flips, `_encryptedURIs` updates, `MetadataUpdated` fires.
5. New owner uses their privkey to unwrap K, then can read+decrypt KV stream and resume agent.

**The judge probe:** "show me the `proof` parameter is non-trivial." Demo plan: deploy our own minimal TEE oracle (or use 0G's reference) — call `transfer()` with a hand-crafted invalid proof, contract reverts. Then call with valid proof, transfer succeeds, new owner reads memory.

**If we skip the oracle and just `transferFrom`, an architect-grade judge will dock us.** The whole differentiation collapses without it.

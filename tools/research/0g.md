3. **The "encrypted intelligence transfer" claim is technically defensible** — but it depends on a **trusted oracle** (TEE or ZKP), not pure on-chain magic. The Solidity `transfer()` takes a `sealedKey` + `proof` parameter; the proof is verified by an oracle contract before ownership flips. Without the oracle, ERC-7857 IS just ERC-721 with an extra hash field. Judges will probe this — be ready to demo the oracle path live.

5. **0G Storage is NOT token-bound by default.** It uses a **KV-on-Log** model with optional `EncryptionKey` (AES-256-CTR client-side). Read access is open; **encryption is the only gate.** This means the iNFT key-rebinding mechanism MUST be in your contract/oracle layer — the storage layer doesn't enforce it for you.
6. **Track B judging artifacts judges grep for:** (a) link to minted iNFT on chainscan-galileo, (b) **proof that intelligence/memory is embedded** (not just a metadata pointer to a public file), (c) <3-min demo video, (d) public GitHub with README + setup, (e) for swarms: explanation of agent comms. 8. **Hackathon timeline:** Open Agents runs **April 24 – May 6, 2026** ($50,000 total). Async/online format. Build through May 3, judge May 6. Our window is now.
9. **The hero "transfer mid-demo, earnings reroute" moment is achievable** — but ONLY if we wire the oracle's TEE re-encryption call into the demo. A pure ERC-721 `transferFrom` will get us disqualified from the iNFT tagline. The reference contract at `0glabs/0g-agent-nft` ships a TEE oracle pattern we can reuse.
10. **Critical risks:** (a) the canonical EIP is still PR-stage, so don't claim "ERC-7857 standard" — say "ERC-7857 draft" or "0G's iNFT spec"; (b) Galileo testnet has had chainId churn (16600 Newton → 16601 Galileo, occasional 16602 references in the wild) — verify in MetaMask before recording demo; (c) 0G Storage encryption is client-side only — if our memory writer uploads plaintext, the "private intelligence" claim breaks.

## 0G Compute

### Endpoints + auth

- **SDK package:** `@0gfoundation/0g-compute-ts-sdk` (TypeScript). Go and Python SDKs also exist via `0g-serving-broker`.
- **CLI:** `0g-compute-cli` — manages providers, sub-accounts, secrets.
- **Web UIs:** `pc.0g.ai` (Advanced mode) and `compute-marketplace.0g.ai/inference`.
- **Auth model:** OpenAI-compatible bearer token. Key format: `app-sk-<SECRET>`. Generated per-provider via `0g-compute-cli inference get-secret --provider <PROVIDER_ADDRESS>`. Each provider has a **wallet-signed sub-account** — users sign requests with their wallet keys.
- **Endpoint discovery:** dynamic per provider via `await broker.inference.getServiceMetadata(providerAddress)` → returns `{ endpoint, model }`. Do NOT hardcode endpoints.

### Available models (mainnet, current)

| Model | Type | Verification | Notes |
|---|---|---|---|
| **GLM-5** | LLM (744B MoE) | **TeeML** | Self-hosted in TEE. Input 1 OG/1M tok, output 3.2 OG/1M tok |
| **DeepSeek Chat v3** | LLM | TeeML | Self-hosted |
| **gpt-oss-120b** | LLM | TeeML | Self-hosted |
| **Qwen3-VL 30B** | Multimodal LLM | TeeML | Vision-capable |
| **Qwen3.6-Plus** | LLM (Alibaba flagship) | **TeeTLS** | Proxied via TEE-attested broker |
| **Whisper Large v3** | Speech-to-text | — | — |
| **z-image** | Text-to-image | — | — |

**TeeML** = model runs inside the TEE itself; responses signed by enclave key. **TeeTLS** = broker runs in TEE and proxies requests to a centralized provider (e.g. Alibaba Cloud Model Studio) over HTTPS, with attestation proving the broker is unmodified.

⚠️ Project context referenced "qwen3.6-plus, GLM-5-FP8" — both are real. GLM-5-FP8 is a quantized variant cited in 0G blog posts; the production listing on the marketplace is "GLM-5". Confirm exact model name via `broker.inference.listService()` before demo.

### Sealed inference / TEE attestation reality

Yes — **attestation is real and verifiable**. SDK exposes:

```ts
const result = await broker.inference.verifyService(
  providerAddress,
  './reports',  // attestation reports written here
  (step) => console.log(step.message)
);
```

Reports save to disk; there's no out-of-the-box "show this hash in a UI" surface, but the report contains the enclave measurement and signed quote — we can hash + render it in our UI ourselves. **For our demo:** display the attestation report digest as a small badge near each agent's reasoning output. This is an artifact judges will respect.

### Hackathon access path

**Open access. No enterprise contract needed.** Steps:
1. Wallet on Galileo → deposit ~5 OG into compute broker.
2. `0g-compute-cli inference list-providers` → pick one running GLM-5.
3. `0g-compute-cli inference get-secret --provider <addr>` → API key.
4. Use OpenAI-compatible client: `baseURL = endpoint`, `apiKey = secret`.

**Rate limits:** 30 RPM sustained per user, 5 burst, 5 concurrent. HTTP 429 on overage. Plenty for our demo flow.

### Verdict for Ledger

**Usable.** Wire `chat.completions` calls to GLM-5 or Qwen3.6-Plus, surface attestation digest in UI. The hero demo runs through this path. **Gotchas:** (1) endpoints are dynamic — query at runtime; (2) the broker handles billing per-call, so funded wallet must be reachable; (3) Qwen3.6-Plus uses TeeTLS not TeeML — slightly weaker trust story, mention TeeML for our hero (GLM-5).

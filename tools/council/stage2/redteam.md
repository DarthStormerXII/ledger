# Stage 2 — Redteam Cross-Critique

## Section 1 — Per-response evaluation

**Response A** got the most important systems-level correction right: the docs misuse the word "atomic" across Base Sepolia settlement and 0G Sepolia reputation updates, and that is exactly the kind of technical overclaim a serious judge can break in one question. It also correctly identifies `ownerOf()` at payment time as the strongest architectural choice, then pushes the team to define mid-flight transfer semantics instead of hiding behind a vague "next earnings" phrase. Its ERC-8004 critique is valuable because the current docs risk claiming standards compliance where the implementation is really a custom rating registry. What it missed is the larger demo-integrity problem: even if the state machine is fixed, the demo still looks fake without visible proof artifacts for AXL, KeeperHub, memory access, and reputation provenance. It may also over-prescribe cutting Base Sepolia; Base USDC is part of the project's current recognizability and the team may be better served by honest eventual-consistency language than by a late settlement-chain pivot. The document-specific changes are mostly correctly stated and would materially improve the spec, but the hidden cost is engineering churn if the team interprets them as mandatory implementation scope rather than as honesty and evidence edits.

**Response B** got the prize-scoring lens right: the problem is not that Ledger lacks sponsor fit, it is that sponsor proof is not packaged in a way a tired reviewer can verify in under a minute. The Proof Matrix, separate `/docs/0g-proof.md`, `/docs/axl-proof.md`, and `/docs/keeperhub-proof.md`, plus visible request IDs, token IDs, node IDs, CIDs, and tx hashes are the highest-output recommendations in the bundle. It also correctly centers the finalist moment on "same worker, same reputation, new owner, next payment lands in the new wallet" instead of generic agent-marketplace mechanics. Where it is weaker is that it underplays the hard iNFT standards critique: a visible before/after table is necessary, but if the table only shows tokenURI continuity, it still does not answer the encrypted-intelligence-transfer challenge. It also suggests spending less time routing every action through KeeperHub, which may conflict with the team's own "all on-chain actions through KeeperHub" claim and can weaken the KeeperHub story unless the reduced scope is disclosed. Its document-specific changes are correctly stated and very actionable; the main hidden cost is that "Evidence Mode" overlays can clutter the product if they are not kept separate from the polished demo UI.

**Response C** is the harshest and most useful integrity attack: it names the exact way the whole submission can be dismissed as an ERC-721 metadata pointer plus a scripted dashboard. It correctly flags the dangerous escape hatches in the docs: simulated KeeperHub reroute UI, cached 0G reasoning replay, fake 47-job history, topology-as-proof, and iNFT memory transfer reduced to a CID. It is also right that the current README/submission copy risks papering over synthetic reputation history and overclaiming "genuinely carries embedded intelligence" without evidence. What it gets wrong or misses is that it stays almost entirely in attack mode and does not distinguish between flaws that require code changes and flaws that can be mitigated by proof packaging or wording discipline. It also underweights Response A's cross-chain atomicity problem, which is a separate technical trap not reducible to "scripted dashboard." The document-specific weaknesses are accurately stated, but applying them literally would mostly reduce overclaiming; the hidden cost is that they do not by themselves create the positive sponsor evidence needed to win.

**Response D** got the demo-consumption reality right: judges experience this through a thumbnail, muted autoplay, the first five seconds, and maybe a 90-second watch cliff, not as a patient viewer of a four-minute cinematic arc. Its recommendation to invert the open and preview the inheritance at frame zero is one of the few changes that could improve finalist odds without touching contracts. It also correctly attacks Shot 2 as ornamental and says the actual iNFT is the data, not a floating crystal. Where it overreaches is by cutting technical language like "ERC-7857" from the emotional peak without making sure the sponsor proof appears somewhere equally visible; silence and numbers are good demo craft, but sponsor judges still need standards evidence. It also ignores several technical landmines that could make the gorgeous cut indefensible in Q&A, especially fake reputation and KeeperHub simulation. Its document-specific changes are strong for the demo script and submission presentation, but they have a hidden cost if the team spends scarce Day 8/9 energy recutting the film while leaving proof artifacts and overclaims unresolved.

## Section 2 — Cross-cutting consensus

**Finding 1: The inheritance moment must be made visually and evidentially undeniable.** B and D both argue that the payment-recipient switch is the finalist moment, while A and C both show that the underlying ownership/payment semantics and memory transfer claims are vulnerable. The concrete document edit is to update `03_DEMO_SCRIPT.md` with a named 2:50 beat showing the same `WorkerINFT`, same reputation, ownerBefore/ownerAfter, and payment recipient switch; update `02_ARCHITECTURE.md` with explicit payment-time `ownerOf()` and mid-flight transfer semantics; and update `07_SUBMISSION_PACK.md` with the same before/after proof table.

**Finding 2: Sponsor claims need proof artifacts, not just polished language.** B asks for a Proof Matrix and sponsor-specific proof docs; C demands forensic evidence for AXL, KeeperHub, 0G memory, and fake-reputation provenance; A asks for hashes, consistency model, and attestation language discipline. The concrete document edit is to add a top-level `Proof Matrix` to the README template in `07_SUBMISSION_PACK.md` and add required evidence bullets to `01_PRD.md` sponsor requirements: token ID/explorer/CID for 0G, node IDs/log excerpts for AXL, KeeperHub request IDs/tx hashes/confirmation times for KeeperHub.

**Finding 3: The current docs overclaim standards and runtime guarantees.** A flags non-atomic cross-chain settlement, loose ERC-8004 language, and unproven sealed inference; C flags ERC-7857 as metadata-pointer theater, simulated reroute risk, and fake reputation history; B independently says proof density is the bottleneck. The concrete document edit is to scrub `00_MASTER_BRIEF.md`, `02_ARCHITECTURE.md`, `03_DEMO_SCRIPT.md`, and `07_SUBMISSION_PACK.md` for claims like "atomically," "verifiable," "genuinely carries embedded intelligence," and "full reputation history," replacing them with evidence-backed wording or explicit limitations.

## Section 3 — Conflicts the team must adjudicate

**Conflict 1: Keep all on-chain actions through KeeperHub or focus KeeperHub on one high-stakes flow.** A says KeeperHub as the only tx execution layer is strategically and architecturally correct; B says routing low-stakes actions through KeeperHub may be engineering drag and the team should prioritize one adverse-condition payment release. The decision criterion is demo evidence by May 2 EOD: if all-actions-through-KeeperHub is already working, keep it because it strengthens the sponsor story; if not, narrow the claim to the flows with real KeeperHub request IDs and tx hashes rather than leaving a broad claim unproven.

**Conflict 2: Technical proof density versus cinematic pacing.** D wants to cut jargon, trim proof exposition, and lead with a silent/visual inheritance hook; B and C want more visible IDs, hashes, logs, and receipts so sponsor judges can verify reality. The decision criterion is audience: the main four-minute video should carry one compact evidence overlay per sponsor beat without becoming a terminal tour, while README and proof docs must hold the full forensic trail. If a proof detail cannot fit cleanly on screen, it must still exist above the fold in the README.
